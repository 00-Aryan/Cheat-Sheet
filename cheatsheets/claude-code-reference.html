You've been using Claude Code like a chat window. This guide will change that. Read once, then keep it open alongside your terminal.

01

Mindset Shift: Chat App vs. Claude Code

Claude.ai (web chat)Claude Code (terminal)
One turn at a timeExecutes multi-step tasks autonomously
You paste code into chatIt reads your actual files directly
You manually apply suggestionsIt writes, edits, and creates files itself
No awareness of your project structureSees your entire directory tree
Forgets everything on refreshPersists configuration via CLAUDE.md

The Agentic Loop

Claude Code doesn't just reply — it acts:

agentic loop
You give a goal
    ↓
Claude reads your files        (Read tool)
    ↓
Claude writes or edits code    (Edit/Write tool)
    ↓
Claude runs the code           (Bash tool)
    ↓
Claude reads output → adjusts → re-runs
    ↓
Reports result to you
The mental model: Think of it as a junior dev sitting next to you. They have read access to your whole project, can run terminal commands, and remember your rules via CLAUDE.md. They make mistakes but iterate fast when corrected. Your job: give clear goals, review outputs, course-correct early.

Essential Slash Commands

Type / on an empty prompt to see all commands in your current setup — built-in, custom, and MCP-provided.

Built-in Commands

/help
Lists all available slash commands — built-in, custom, and MCP-sourced.
ML: Run first when stuck. Check before repeating yourself.
/clear
Wipes conversation history and resets context window. Fresh start.
ML: Use when switching from debugging train.py → new task
⚠ Loses all session history. Use /compact if you want a summary preserved.
/compact must-know
Summarizes older messages to reclaim context space while preserving key information.
ML: Run proactively in long debugging sessions — before Claude forgets early decisions.
/context  or /usage
Shows current context window usage and when auto-compaction triggers.
→ "You've used 68% of context. Auto-compact at 95%."
/memory must-know
Opens your global CLAUDE.md for editing. Persistent across all sessions + all projects.
ML: Add "Always use type hints" or "I use conda, not pip" once. Done forever.
/add-dir  (or --add-dir flag)
Grants Claude access to directories outside your current working directory.
claude --add-dir ../shared-utils --add-dir ../datasets
/doctor
Health check: Node version, auth status, tool connectivity, environment variables.
Run this FIRST when something breaks unexpectedly.
/model
Switches the underlying Claude model for current session.
Haiku → fast/cheap. Sonnet → most tasks. Opus → complex planning.
/ide
Connects to VS Code or JetBrains. Git diffs open in editor with syntax highlighting.
ML: Essential for reviewing model file changes with side-by-side diffs.
/terminal-setup
Configures Alt+Enter (or Option+Enter on Mac) to create a new line instead of sending.
Run once → restart terminal → write multi-line prompts without accidents.
/permissions
Shows what Claude can read, write, and execute in current session.
Review before automated pipelines touch your training data.
/mcp
Lists and manages connected MCP servers (external integrations).
Connect DB MCP → Claude queries your SQL tables directly.
/btw pro
Ask a side question while Claude is actively working — without interrupting it or touching history.
/btw what's the default lr in SGDClassifier?
/intro
Scans your project and generates a starter CLAUDE.md file automatically.
Run once per new project. Then edit to add your data safety rules.
/agents pro
Shows and manages running sub-agents (specialized Claude instances with their own context).
Use when delegating tasks: one agent for data, one for modeling.
/vim
Toggles vim keybinding mode for the input field.
Comfortable with vim? This makes writing long prompts much faster.

Custom Slash Commands

Store commands as Markdown files — they become typed shortcuts.

LocationScopeUse for
.claude/commands/cmd.mdCurrent project onlyProject-specific workflows (shared via git)
~/.claude/commands/cmd.mdAll projectsPersonal command library
.claude/skills/Current projectModern approach — natural language invokable
~/.claude/skills/All projectsPersonal skills library
bash — create ML profiler command
mkdir -p .claude/commands
cat > .claude/commands/profile-model.md << 'EOF'
---
description: Profile model training step for bottlenecks
allowed-tools: Bash, Read
---
Profile the model training script for performance bottlenecks:
1. Check memory usage with tracemalloc or memory_profiler
2. Identify slow operations using cProfile
3. Suggest optimizations (vectorization, batching, caching)
4. Generate a summary report

Target file: $ARGUMENTS
EOF

Usage: /profile-model src/train.py

Rule of thumb: Any prompt you use more than twice becomes a custom command. Repetitive prompts are technical debt.
03

CLAUDE.md: Your Project's Brain

CLAUDE.md is a Markdown file Claude reads at the start of every session. It's your project's persistent instruction set — ground truth for how Claude behaves in your codebase.

FileScopeUse for
~/.claude/CLAUDE.mdGlobal — all projectsPersonal preferences, tool habits, general rules
./CLAUDE.md (project root)This project onlyProject structure, conventions, data warnings
.claude/rules/*.mdPath-specificRules that only activate on matching files
Keep CLAUDE.md under 200 lines. Every word loads on every message — it costs context tokens. Move large docs to skills files (loaded on demand only).

ML/DS CLAUDE.md Template

markdown — copy-paste into your project root
# Project: [Your ML Project Name]

## Environment
- Python 3.11 via conda — environment name: `ml-env`
- Activate with: `conda activate ml-env`
- Install packages: `pip install package` OR `conda install package`
- Do NOT use bare `pip install` — check requirements.txt first

## Project Structure
project/
├── data/
│   ├── raw/          ← NEVER modify. Read-only source of truth.
│   ├── processed/    ← Cleaned, feature-engineered data
│   └── splits/       ← Train/val/test CSVs
├── notebooks/        ← Exploration only. Not production code.
├── src/
│   ├── data/         ← Data loading and preprocessing
│   ├── features/     ← Feature engineering
│   ├── models/       ← Training and evaluation
│   └── utils/        ← Shared utilities
├── tests/            ← pytest unit tests
├── models/           ← Saved artifacts (.pkl, .pt, .joblib)
├── reports/          ← Figures and outputs
└── requirements.txt

## Critical Rules
- NEVER overwrite data/raw/ without explicit confirmation.
- Always ask before deleting any file.
- Do not commit model weights or large data files to git.

## Code Conventions
- Type hints on all function signatures
- Google-style docstrings on every function
- Functions under 40 lines — split if longer
- Use sklearn.pipeline.Pipeline, not manual chaining
- Prefer pathlib.Path over os.path
- Use logging module, not bare print()

## ML-Specific Conventions
- Set SEED = 42 at top of every training script
- Always stratify splits: train_test_split(..., stratify=y)
- Log metrics to console AND save to reports/metrics.json
- Never hardcode hyperparameters — use config dict or argparse

## Common Commands
- Train: python src/models/train.py --config configs/default.yaml
- Preprocess: python src/data/preprocess.py
- Test: pytest tests/ -v
- Validate data: python src/data/validate.py

## Current Status
- Working on: [update this manually]
- Known issue: [update this manually]

Add to Memory Instantly (Without Opening the File)

Type # followed by any statement in the REPL to add it to your project CLAUDE.md immediately:

in-session memory shortcut
# Always use StandardScaler before PCA in this project
# The target variable is 'converted', not 'label'
# Model checkpoints go in models/checkpoints/, not models/

Rewind Mode (Escape × 2)

One of the most underused features. Press Escape twice, scroll back with ↑, press Enter on any point, then choose:

  • Restore code and conversation — full rollback
  • Restore conversation — rewind chat, keep code changes
  • Restore code — revert files, keep conversation
  • Summarize from here — compact from this point forward
ML use case: Claude refactored your feature engineering in a direction you hate. Rewind to before it started, correct the instruction, try again.

Stash Workflow (Ctrl+S)

scenario
You're typing:  "Now refactor the preprocess function to..."
You realize:    "wait — I need to check which columns are null first"

→ Press Ctrl+S  (stash current prompt)
→ Type:  ! python -c "import pandas as pd; print(pd.read_csv('data/raw/train.csv').isnull().sum())"
→ Review output
→ Your original prompt auto-restores in the input field
→ Continue writing it with full context

Persistent Sessions with tmux

bash
# Start a named session
tmux new -s ml-session

# Inside tmux — launch Claude Code
claude

# Detach (session keeps running even if terminal closes)
Ctrl+B, then D

# Re-attach later from any terminal
tmux attach -t ml-session
Why this matters: If Claude is preprocessing a 10GB dataset and you close your laptop, tmux keeps it alive. Non-negotiable for long jobs.

Workflow Recipes for the AI/ML Student

Recipe A Starting a New ML Project from Scratch
1
Launch Claude Code in your new project directory.
cd ~/projects && mkdir purchase-predictor && cd purchase-predictor && claude
2
Orient Claude immediately with full context before asking it to act.
"I'm building a binary classification model to predict e-commerce purchase intent. Dataset: ~100k rows, tabular session features. Stack: Python, scikit-learn, pandas. Set up the full project structure."
3
After structure is created, generate CLAUDE.md: /intro → then edit to add your critical rules (data safety, random seeds, etc.)
4
Add global personal preferences: /memory
5
Initialize git immediately: ! git init && git add . && git commit -m "Initial project structure"
Key insight: Structure first, code second. Claude will use the structure it creates as a map for all future work in that session.
Recipe B Adding a RAG Retrieval Component to an Existing Codebase
1
Give Claude full access to the project and document store.
claude --add-dir . --add-dir ../data/documents
2
Orient before asking. "Read the README and src/ directory. Tell me what you found before we proceed."
Wait for the summary. If Claude misunderstands — correct it NOW before it writes a single line.
3
Specify the task with precision. "Add retrieval using FAISS + sentence-transformers. Documents in ../data/documents/*.txt. Create: src/retrieval/retriever.py with DocumentIndexer and Retriever classes. Write tests in tests/test_retriever.py."
4
Run tests immediately: ! pytest tests/test_retriever.py -v
5
If tests fail, show the full error. "Tests failed with: [paste output]. Fix the root cause — don't just suppress the error."
Recipe C Refactoring a Messy Notebook into Clean Python Scripts
1
Export notebook to script first.
! jupyter nbconvert --to script notebooks/data_exploration.ipynb
2
Give Claude the refactor spec. "Read notebooks/data_exploration.py. It's a converted Jupyter notebook — messy. Refactor into src/data/preprocess.py. Requirements: typed functions with docstrings, no bare print() (use logging), separate load/preprocess functions, CLI-callable main(), pathlib for all paths."
3
Verify with a real run. "Run the refactored script on data/raw/sample.csv and show me shape and dtypes before/after."
4
Write tests. "Write pytest unit tests using a small synthetic DataFrame as fixture data."
Recipe D Debugging a Training Script (Accuracy Plateauing)
1
Share full context — not just the symptom. "Read src/models/train.py and reports/metrics.json. Accuracy plateaus at 61%. I've tried: adjusting LR, adding dropout. Tell me what you think is happening before suggesting fixes."
This "tell me before suggesting" step is critical — it surfaces misunderstandings before code is written.
2
Run diagnostics.
! python src/models/train.py --debug --epochs 5
3
Inspect specific outputs (confusion matrix, per-class precision/recall).
4
Iterate one change at a time. "Propose exactly one change, explain why it should help, then implement it. Don't change multiple things at once — I need to know what worked."
Anti-pattern: "Fix the model." → Claude makes 12 changes simultaneously and you learn nothing.

Best Practices & Anti-Patterns

✓ Do This

Orient Claude before asking it to act — "Read these files. Tell me what they do. Then I'll tell you what to change."
Give Claude the whole project via --add-dir. Full context → fewer hallucinations.
Break tasks into named steps. "Step 1: Create loader. Test it. Step 2: ..."
Run tests after every change. ! pytest tests/ -v becomes a reflex.
Commit after every working increment. ! git add . && git commit -m "feat: ..."
Correct early. If Claude misunderstands on the first response, fix it before it builds 200 lines on a wrong assumption.
Use ! for shell commands. Saves tokens, runs faster, no narration.
Keep CLAUDE.md under 200 lines. Dense prose wastes context every session.

✗ Don't Do This

Paste raw code into the prompt. Claude reads your files directly — --add-dir is free.
Ignore /compact in long sessions. Above 70% context → information is already being compressed.
Treat it like a search engine. "What's the best embedding model for RAG?" → web search. "Read my retriever.py and suggest one given my constraints" → Claude Code.
Ask it to do everything at once. "Build the entire ML pipeline end-to-end" → untestable, untraceable mess.
Skip Claude's plan review. If it says something wrong before acting — correct it. Proceeding on a wrong assumption is the #1 source of wasted sessions.
Give vague data permission. CLAUDE.md must say "Never modify data/raw/ without confirmation." Make it a rule, not a hope.
Carry unrelated context. /clear between different tasks — morning debugging context will confuse afternoon feature work.

Quick-Start Checklist

  • Install Claude Code: curl -fsSL https://claude.ai/install.sh | bash
  • Authenticate: claude auth login
  • Run /doctor to verify setup
  • Run /terminal-setup and restart terminal (enables Alt+Enter for newlines)
  • Set global preferences: /memory → add language, tool, and style preferences
  • For each new project: run /intro to generate starter CLAUDE.md
  • Edit CLAUDE.md to add: data safety rules, project structure, common test commands
  • Create your first custom command in .claude/commands/
  • Install tmux for persistent sessions: tmux new -s project-name
  • Set --add-dir as a habit — always give Claude your full project on launch
